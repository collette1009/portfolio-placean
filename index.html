CASE 01 / 04

400여 제휴사의 상이한 예약·취소·위약 정책을 공통 운영 기준으로 표준화

예약 운영제휴 골프장정책 표준화변경 관리
내 역할 · 400여 곳 어드민 전수 점검 · 공통 속성·조회 순서 정의 · 골프장 예약실 건별 확인 · 누락 정책 보완 · FAQ·온보딩 업데이트 · 예약 문의 운영

문제

고객이 예약·취소·위약·동반자 변경을 문의하면 해당 골프장 정책과 예약 상황을 건별로 확인해야 했습니다. 그러나 400여 제휴 골프장의 운영시간, 취소 기한, 위약 적용 범위, 동반자 제한 정보는 위치와 표현이 제각각이었고 일부는 누락되거나 모호했습니다. 고객 문의를 해결하는 과정에서 파트너 확인이 반복됐지만, 확인 결과가 동일한 방식으로 축적되지 않으면 담당자마다 검색 순서와 판단이 달라질 수 있었습니다.

판단과 실행

A
고객 상황을 정리해 파트너에 건별 확인
고객 예약 정보, 요청사항, 취소 시점과 필요한 조치를 먼저 정리한 뒤 해당 골프장 예약실에 직접 확인하고, 확인 결과와 가능한 다음 조치를 고객에게 안내했습니다.
B
반복 확인 정보를 공통 속성으로 정의
예약실 연락처, 운영시간, 취소 가능 기한, 위약 적용 범위, 위약금 입금 계좌, 동반자 제한 여부를 공통 필드로 정의했습니다.
C
조회 순서·표현·누락 확인 기준 표준화
운영자가 위에서부터 같은 순서로 확인할 수 있게 정보 위치와 문장을 통일하고, 누락·불명확한 내용은 추정하지 않고 파트너에 재확인해 보완했습니다.
D
확인 결과를 운영 지식으로 환류
복합·예외 케이스와 정책 변경 내용을 어드민·FAQ·상담 가이드·온보딩 자료에 함께 반영했습니다.

결과

건별 문의 해결 과정에서 반복 확인되던 파트너 정보를 400여 곳 공통 속성과 조회 순서로 전환했습니다. 고객 문의가 들어올 때마다 처음부터 찾는 구조에서, 파트너별 차이는 유지하되 동일한 방식으로 확인·업데이트할 수 있는 운영 지식 구조로 바꿨습니다.

남긴 운영 기준

파트너 운영은 문의를 대신 전달하는 일이 아니라, 건별 확인 결과를 재사용 가능한 기준으로 축적하고 변경 시점까지 관리하는 일입니다. 파트너 확인과 운영 정책이 연결돼야 파트너 수가 늘어도 실행 품질을 유지할 수 있습니다.

CASE 02 / 04

3PL 전환 초기 4,789건의 이슈와 데이터 공백을 진단해 임시 운영·시스템 개선·자동화로 연결

3PL 전환데이터 공백임시 운영자동화
내 역할 · 초기 상황 확인 · 처리 기준 확정 · 이슈 수치화·유형화 · 데이터 공백 정의 · 리소스·일정 산정 · 임시 운영·CRM 개선 · 자동화 · 계약 기반 월별 피해 정산

문제

3PL 전환 직후 오피킹·포장불량·출고누락·미출고가 동시에 발생했지만 각 이슈가 개별 CS 문의로만 남아 전체 규모를 알기 어려웠습니다. 미출 리스트가 사전 공유되지 않거나 송장번호가 주문정보와 연결되지 않았고, CRM 출고리스트에는 보상 자료 가공에 필요한 회원코드가 없어 후속 업무가 주문별 수기 조회에 의존했습니다.

판단과 실행

A
초기 상황 확인·처리 기준 확정
출고 여부와 처리 계획이 확정되지 않은 상황에서 전량·부분 미출 여부, 미출 리스트, 당일 처리 계획과 신규 주문 출고 일정을 확인 항목으로 정리했습니다. 확정 전에도 지연 안내가 필요하다고 판단해 선제 안내 방향을 합의했고, 이후 도착희망일과 출고 상태별로 취소·재배송·보류 기준을 구분해 경영진과 확정했습니다. 2/18~3/9 누적 4,789건은 오피킹 47건, 포장불량 55건, 출고누락 800건, 미출고 3,887건으로 유형화했습니다.
B
데이터 공백 정의·구조 개선 제안
미출 리스트 미공유, 송장번호 미연결, 회원코드 미노출이 고객 안내·배송조회·보상 자료·재배송 처리에 미치는 영향을 정리했습니다. 고객 문의 이후에야 확인하는 구조를 줄이기 위해 정확한 미출 리스트의 사전 공유와 주문번호–운송장번호 시스템 연결을 제안했습니다.
C
시스템 정상화 전 임시 운영
송장 연결이 정상화되기 전에는 3PL 파트너의 운송장 파일을 가공해 배송조회 링크를 생성하고 개별 상태를 확인했습니다.
D
CRM 개선
보상 대상 자료 가공 시 주문별 조회가 필요했던 문제를 공유하고, CRM 출고리스트에서 회원코드를 직접 확인하도록 개선을 요청해 반영을 확인했습니다.
E
대량 처리 자동화
Chrome Extension·Apps Script로 재배송 등록을 6→2단계로 단축해 전환 초기 재배송 1,151건을 처리하고, 잔여 취소 대상과 처리 일정을 다시 산정했습니다. 이후 반품 등록은 7→1단계로 단축해 도입 이후 발생한 반품 등록 전 건에 상시 적용했습니다.
F
계약 근거 발굴·월별 온라인 피해 정산
전환 초기 4,789건은 도입 단계의 일회성 진단 수치로 분리하고, 이후 반복 발생하는 온라인 피해는 별도의 월별 정산 항목으로 관리했습니다. 출고누락·오피킹·포장불량·배송 사고로 회사가 선처리한 비용이 회수되지 않는 문제를 확인해 계약서의 상차 후·운송 중 책임 조항을 검토하고, 환불·재배송·고객 보상 등 확인 가능한 피해를 3PL 파트너 물류비 차감 항목으로 반영하는 기준을 대표에게 제안했습니다. 승인 이후 월별 확정액을 산정하고, 귀책이 불명확한 건은 제외하며 파트너 이의·중복 건은 주문번호 단위로 재검토해 차감 반영 자료로 관리했습니다.

결과

전환 초기 이슈 규모와 유형을 수치로 확인할 수 있게 했고, 시스템이 완전히 정상화되기 전에도 고객 안내와 후속 처리가 멈추지 않는 임시 운영방안을 마련했습니다. CRM 출고리스트에서 회원코드를 직접 확인할 수 있도록 개선했으며, 재배송 1,151건을 처리하고 이후 월별 피해현황을 같은 기준으로 추적할 수 있는 구조로 연결했습니다. 재배송 선행 처리 후 잔여 물량과 처리 리스크를 다시 계산해 필요한 지원 인원과 완료 일정을 조정했습니다. 계약 검토에서 시작한 제안은 대표 승인 후 월별 온라인 피해 확정액을 산정하는 반복 운영 역할로 정착됐습니다.

남긴 운영 기준

신규 운영 구조의 도입 단계에서는 일회성 진단 수치와 반복 관리 지표를 구분해야 합니다. 시스템 개선이 완료되기 전까지 운영을 유지할 임시 방안과 장기 개선안을 함께 설계해야 실제 안착으로 이어집니다.

CASE 03 / 04

지원 인력과 매장이 담당자에 따라 다르게 판단하지 않도록 운영 기준·검수·안내 체계 정착

운영 기준답변 검수에스컬레이션온보딩
내 역할 · 문의 유형·처리 범위 정리 · 검수·에스컬레이션 기준 문서화 · FAQ·SOP·매크로 정비 · 온보딩 자료 제작 · 답변 검수·피드백 운영

문제

회생 절차와 조직 변동으로 기존 운영 인력이 순차 이탈하고, CS 경험이 없는 타부서 지원 인력이 고객 응대 업무에 투입돼야 했습니다. 기존 문서와 실제 처리 기준이 분산돼 있어 담당자마다 확인 범위와 판단이 달라질 가능성이 있었습니다.

판단과 실행

A
문의 유형과 처리 범위 재정리
인입되는 문의를 유형별로 나누고, 지원 인력이 바로 안내할 수 있는 범위와 매장·상품·물류·개발 확인이 필요한 범위를 구분해 처리 기준을 정리했습니다.
B
검수·에스컬레이션 기준 문서화
답변 전 확인해야 할 항목과 판단이 어려운 경우의 에스컬레이션 경로·담당자를 문서로 정리해 개인 판단에만 의존하지 않도록 했습니다.
C
FAQ·SOP·매크로 정비
문의 유형별 FAQ와 SOP, 매크로를 정비해 지원 인력이 같은 기준으로 빠르게 답변할 수 있게 했습니다.
D
온보딩과 피드백 운영
신규 투입 인력에게 온보딩 자료로 처리 기준을 안내하고, 실제 답변을 검수해 반복되는 오류에 피드백을 제공했습니다.
E
매장 품질·민감 이슈 대응 기준 수립
과거 조치가 완료된 품질 이슈가 외부 기사로 다시 언급되면서 매장별 고객 안내 기준이 필요했습니다. 실제 발생 시점과 조치 완료 여부를 확인하고, 고객에게 안내 가능한 사실과 내부 확인 사항을 구분했습니다. 매장이 먼저 설명할 내용과 고객이 직접 문의한 경우에만 답변할 범위를 나누어 표준 문구를 제시하고, 경영진 확인을 거쳐 실행 방향을 정렬했습니다.

결과

운영이 특정 담당자의 숙련도에만 의존하지 않도록 기준을 정착시켰습니다. 평균 처리시간은 3.1분에서 1.7분, 매크로 사용률은 10%에서 62%로 개선됐고, 온보딩 기간은 10일에서 4~5일로 단축됐습니다.

남긴 운영 기준

운영 기준은 문서를 만드는 데서 끝나지 않고, 실제 검수·피드백·온보딩을 통해 누구나 같은 판단을 할 수 있을 때 현장 실행 구조로 정착됩니다.

CASE 04 / 04

반복 문의를 주문·재고·화면 상태와 비교해 서비스 구조 문제로 정의하고 당일 개선

VOC 패턴운영 데이터개발 협업서비스 개선
내 역할 · 신규 상품 등록 직후 이상 문의 포착 · 문의·주문·재고·화면 비교 · 공통 발생 조건 정의 · 최소 개선안 제안 · 개발 협의 · 반영 후 문의 추이 확인

문제

화요일에만 도착 가능한 신규 상품 등록 직후 동일 문의가 6~10건 발생했습니다. 배송 가능 요일이 다른 상품을 함께 담는 경우 주문 과정에서 제한이 발생했지만, 화면에는 일반적인 품절형 메시지가 노출돼 고객이 어떤 상품의 배송 조건 때문에 제한되는지 알기 어려웠습니다.

판단과 실행

A
문의·주문·재고·화면 비교
문의 발생 시점, 고객 장바구니 구성, 실제 재고, 상품별 배송 조건과 화면 메시지를 비교해 실제 품절이 아니라 배송 조건 조합에서 발생하는 공통 문제임을 특정했습니다.
B
전체 로직이 아닌 고객 혼선 지점 개선
주문 제한 로직 전체를 변경하기보다 고객이 막히는 장바구니 화면에 상품별 배송 가능 요일과 나누어 주문이 필요할 수 있다는 정보를 노출하는 최소 수정안을 정리했습니다.
C
개발 협업·확장 적용
CTO·개발 담당자와 구현 범위를 협의해 당일 반영하고, 같은 구조를 토요일 도착 전용 상품에도 확장했습니다.

결과

등록 당일 6~10건 발생한 관련 문의는 반영 이후 확인 기간 내 0건 수준으로 감소했습니다. 고객이 상담을 요청하기 전에 상품 조건과 필요한 행동을 구매 화면에서 이해할 수 있게 했습니다.

반영 결과 화면
배송 조건이 다른 상품의 안내 구조상품별 배송 가능 요일과 나누어 주문이 필요할 수 있음을 장바구니에서 확인하도록 했습니다.

남긴 운영 기준

반복 문의는 단순 VOC가 아니라 서비스가 실제 사용되는 과정의 마찰 지점입니다. 문의가 발생한 시점과 조건을 운영 데이터와 함께 보면, 기획·개발 조직이 실행할 수 있는 개선 과제로 바꿀 수 있습니다.

플레이스앤에서 운영을 정착시키는 방식

01 도입 기준 구조화
매장 유형·정책·설정값·예외 조건을 공통 항목으로 정리합니다.
02 안착 이슈 관리
발생 조건·영향 범위·재현 정보와 안내·이관 기준을 정리합니다.
03 개선 과제 연결
교육·정책·상품 정보·시스템 문제를 구분해 담당 조직과 완료 기준을 맞춥니다.
04 확장과 자동화
반복 이슈와 처리시간을 운영 지표로 확인하고 다음 매장 도입에 재사용합니다.

예약 플랫폼에서 400여 파트너의 운영 정책과 확인 기준을 표준화했고, 이커머스에서는 신규 물류 구조의 도입·안착, 매장 안내 기준, 데이터 공백과 반복 업무를 운영 기준·시스템 개선·자동화로 연결했습니다. 플레이스앤의 예약·웨이팅·매장 솔루션 구조를 빠르게 익히고, 매장별 차이를 유지하면서도 확장 가능한 운영 기준으로 만들겠습니다.
